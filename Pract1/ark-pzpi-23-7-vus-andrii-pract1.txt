2 ЗАВДАННЯ

Студент повинен обрати мову програмування, дослідити основні рекомендації для написання якісного коду у цій мові, продемонструвати ці рекомендації на прикладах коду, обґрунтувати кожну рекомендацію і надати детальні пояснення. 
Студенту необхідно обрати мову програмування для аналізу. Після вибору мови програмування студенту потрібно ознайомитися з основними рекомендаціями для написання якісного та чистого коду у цій мові. Це можуть бути стильові рекомендації, правила найменування змінних, функцій та класів, структура коду, принципи рефакторингу, оптимізація продуктивності, обробка помилок, дотримання парадигм програмування, тестування та документування коду тощо.
Студент повинен підготувати один або кілька прикладів коду, які демонструють, як потрібно застосовувати наведені рекомендації. Приклад коду може бути загальним і об'єднувати декілька рекомендацій, або ж студент може підготувати кілька окремих прикладів для різних аспектів рекомендацій.
Студент повинен надати докладні пояснення щодо кожної рекомендації, підготувати слайди або текстову доповідь для презентації власного аналізу та представити свою доповідь викладачу.
 
3 ОПИС ВИКОНАНОЇ РОБОТИ

3.1 Вибір мови програмування
Для аналізу конвенцій кодування обрано мову програмування Java – сучасну, об'єктно-орієнтовану мову програмування, розроблену Sun Microsystems. Java є однією з найпоширеніших мов для розробки корпоративних, веб- та мобільних застосунків, що робить стандартизацію коду критично важливою.

3.2 Важливість конвенцій кодування
Дотримання конвенцій кодування є критичним для життєвого циклу програмного забезпечення. Близько 80% витрат на програмне забезпечення протягом його життєвого циклу припадає на підтримку. Гарно написаний, узгоджений код значно полегшує виправлення помилок та додавання нових функцій. Єдиний стиль кодування дозволяє різним розробникам ефективно працювати над одним проєктом, зменшуючи час на розуміння чужої логіки.

3.3 Джерела конвенцій
Багато компаній та різних розробників створювали власні стандарти написання коду Java. Але для даної роботи було вирішено опиратися на стандарти кодування для Java, що були розроблені та опубліковані компанією Sun Microsystems, оригінальним творцем мови. Ці рекомендації, викладені у документі "Java Code Conventions", стали де-факто головним стандартом в індустрії. Після придбання Sun компанією Oracle, ці конвенції продовжують підтримуватись та розвиватись.

3.4 Правила іменування
Ідентифікатори у Java повинні бути змістовними та відповідати чітким правилам. Пакети іменуються у lowercase. Класи та Інтерфейси використовують PascalCase. Методи та змінні використовують camelCase. Константи що є static final, іменуються у UPPER_SNAKE_CASE. Нижче наведено приклади коду з порушенням та дотриманням цих правил.

1  // Погано
2  class myclass {
3      public static final int min_width = 4;
4      int I; // Неясно, що це
5      void RUN() { ... }
6  }
    
1  // Добре
2  class MyClass {
3     public static final int MIN_WIDTH = 4;
4     int loopCounter; // Змістовне ім'я
5     void runCalculation() { ... }
6  }

3.5 Форматування та Структура коду
Форматування забезпечує візуальну структуру коду. Довжина рядків не повинна перевищувати 80 символів, хоча в сучасних практиках допускається до 100-120. Це запобігає горизонтальному прокручуванню. Для відступів використовується 4 пробіли. Використання символів табуляції не рекомендується. Розташування фігурних дужок слідує стилю K&R,  тобто відкриваюча дужка { розміщується в кінці того ж рядка, що й оголошення. Закриваюча дужка } розміщується на новому рядку, на тому ж рівні відступу, що й її структура.

1  // Погано
2  class Sample
3  {
4      public static void main(String args[])
5      {
6          if (condition1 || condition2 || condition3 || condition4) 
7  { ... }
8      }
9  }
 
1  // Добре 
2  class Sample {
3      public static void main(String args[]) {
4          if (condition1 || condition2
5                  || condition3 || condition4) {
6              ...
7          }
8      }
9  }

3.6 Структура файлів
Кожен *.java файл повинен мати чітку структуру та містити не більше одного public класу. Файли довжиною понад 2000 рядків є громіздкими та їх слід уникати. Структура файлу повинна бути наступною, у вказаному порядку:
1.	Заголовок файлу (File Header) з початковим коментарем;
2.	Оголошення package;
3.	Оголошення import, згруповані;
4.	Оголошення public класу (включаючи Javadoc класу).

1  /*
2   * 1. Заголовок файлу (File Header)
3   * (наприклад, коментар про ліцензію або копірайт)
4   */
5  
6  // 2. Оголошення package
7  package com.example.project;
8  
9  // 3. Оголошення import
10 import java.util.List;
11 import java.util.ArrayList;
12 
13 /**
14  * 4. Javadoc класу
15  */
16 public class FileStructureExample {
17     // ... (вміст класу)
18 }

3.7 Коментарі та Документація
У Java існує два типи коментарів. Коментарі реалізації (Implementation Comments) використовуються для коментування коду всередині методів. Вони пояснюють, чому була обрана певна реалізація, або як працює складний алгоритм. Документаційні коментарі (Documentation Comments), або Javadoc, використовуються для опису API. Вони розміщуються перед оголошеннями класів, полів, конструкторів та методів. Javadoc використовує теги, як @param для опису параметрів, @return для опису поверненого значення, та @throws для опису винятків.

1  /**
2   * Розраховує заробітну плату для працівника.
3   *
4   * @param employee Об'єкт працівника для розрахунку.
5   * @return Розрахована заробітна плата.
6   * @throws IllegalArgumentException якщо статус працівника недійсний.
7   */
8  public double calculatePay(Employee employee) throws IllegalArgumentException {
9      // Коментар реалізації: перевіряємо статус перед розрахунком
10     if (employee.getStatus() == null) {
11         throw new IllegalArgumentException("Invalid employee status");
12     }
13     // ... логіка розрахунку ...
14 }

3.8 Оголошення
Конвенції регулюють порядок та форматування оголошень. Оголошення локальних змінних слід розміщувати на початку блоку коду, до їх першого використання. Не рекомендується оголошувати змінні різних типів або кілька змінних в одному рядку. При оголошенні масивів, квадратні дужки [] повинні бути частиною типу, а не імені змінної.
1  // Погано 
2  int count, size;
3  String nameList[];
  
1  // Добре 
2  int count;
3  int size;
4  String[] nameList;

3.9 Створення класів
Порядок оголошень всередині тіла класу є важливим для читабельності. Рекомендований порядок:
1.	Статичні змінні, спочатку public, потім protected, private;
2.	Змінні екземпляра (instance variables), у тому ж порядку видимості;
3.	Конструктори;
4.	Методи, згруповані за функціональністю.

1  // Погано (порядок порушено)
2  class MixedExample {
3      // 4. Метод
4      public void doSomething() { ... } 
5      
6      // 2. Змінна екземпляра
7      private int instanceVar; 
8      
9       // 3. Конструктор
10     public MixedExample() { ... } 
11     
12     // 1. Статична змінна
13     public static final int MAX_USERS = 10; 
14 }

1  // Добре (згідно з конвенцією)
2  class ClassStructureExample {
3      // 1. Статичні змінні
4      public static int staticVar1;
5      private static Object staticVar2;
6      
7      // 2. Змінні екземпляра
8      public Object instanceVar1;
9      protected int instanceVar2;
10     
11     // 3. Конструктор
12     public ClassStructureExample() {
13         // ...
14     }
15     
16     // 4. Методи
17     public void doSomething() {
18         // ...
19     }
20 }

3.10 Загальні практики
Змінні екземпляра не повинні оголошуватись як public. Доступ до них повинен надаватися через public методи (accessor methods: get, set), що дозволяє контролювати та валідувати значення.

1  // Погано (прямий доступ)
2  class Car {
3      public int speed; 
4  }

1  // Добре (інкапсуляція)
2  class Car {
3      private int speed;
4      public int getSpeed() {
5          return speed;
6      }
7      public void setSpeed(int speed) {
8          if (speed >= 0) { // Валідація
9              this.speed = speed;
10         }
11     }
12 }

Дужки повинні використовуватися для групування виразів, щоб зробити код більш читабельним та уникнути плутанини з пріоритетом операторів.

1  // Погано (неоднозначно)
2  if (a == b && c == d || a == e) { ... }

1  // Добре (пріоритет очевидний)
2  if ((a == b && c == d) || (a == e)) { ... }

Слід уникати використання числових констант безпосередньо в коді. Для них необхідно створювати іменовані константи.

1  // Погано
2  for (int i = 0; i < 10; i++) { ... }

1  // Добре
2  static final int MAX_ITERATIONS = 10;
3  for (int i = 0; i < MAX_ITERATIONS; i++) { ... }

При використанні тернарного оператора (? :), вираз зліва від ? повинен бути простим. Складні вирази слід виносити в окремі змінні.

1  // Погано (складний вираз)
2  result = (a == b && c == d) ? x : y;

1  // Добре (простий вираз)
2  boolean condition = (a == b && c == d);
3  result = condition ? x : y;

Спеціальні маркери повинні використовуватися для коду, який є тимчасовим або потребує уваги. Наприклад, //ХХХ для коду, який неправильний, але працює.

1  // XXX: Це тимчасове рішення.
2  // Необхідно провести рефакторинг.
3  int temporaryValue = someLegacyMethod();



 
ВИСНОВКИ

У результаті виконання практичного заняття було вивчено основні рекомендації для написання коду мовою програмування Java, створені Sun Microsystems. 
Було розглянуто основні конвенції та правила, які регулюють стилі відступів, створення класів та файлів, іменування змінних, написання коментарів тощо. На основі цього було створено презентацію (Додаток Б), яка має на меті закріпити власні знання, а також поділитися здобутою інформацією з іншими.
Виконана робота показала, що послідовне дотримання офіційних стандартів оформлення коду є необхідною умовою для професійної розробки програмного забезпечення. Впровадження цих практик дозволяє створювати якісніший, надійніший та більш масштабований код, що полегшує розуміння, підтримку та командну взаємодію.
